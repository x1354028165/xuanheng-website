一、现有官网诊断

1.1 官网的战略定位与角色

官网服务两类用户

旭衡官网需要同时服务两类用户，在不同阶段扮演不同角色：

面向潜在客户：销售漏斗的顶层入口
陌生访客进入官网 ↓ 快速理解旭衡是谁、能解决什么问题（品牌建立信任） ↓ 找到与自己场景匹配的解决方案（精准分流） ↓ 产生咨询/申请意向，进入销售流程（线索转化）

旭衡目标客户差异大（集成商、研发型企业、VPP运营商），诉求各不相同，官网必须能够精准分流，而不是让所有人看同一套内容。

面向已有客户：服务自助的统一入口
客户成交后 ↓ 官网作为持续服务门户：文档下载、API接入、固件更新、故障报修 ↓ 减少人工支持成本，提升客户自助效率

现有定位语与三个主张

现有官网的定位语是：

"跨品牌能源管理解决方案的优质选择"
旭衡电子（深圳）有限公司拥有成熟、稳定的开发能力。优秀的专业研发团队致力研究 EMS + 边缘计算智能网关、新能源智慧管理云平台，为您提供完整、多场景全面产品解决方案，适用于微电网、户用、充电站、工商储等各种场景的能源管理。

这段定位语传达了三个核心主张，本身是准确的：

主张原文对应
跨品牌"跨品牌能源管理解决方案"
软硬一体"EMS + 边缘计算智能网关 + 云平台"
多场景覆盖"微电网、户用、充电站、工商储"

五大核心职责（评判标准）

将两类用户的需求与三个定位主张结合，官网需要履行五大核心职责。以下诊断将以此为标准：

职责说明当前状态
品牌背书专业形象建立，让初次接触的客户建立信任⚠️ 内容陈旧，合作品牌展示关闭
客户分流不同类型访客找到各自的路径和转化动作❌ 无分流设计，所有人看同一套内容
技术能力证明让客户快速判断"你们能解决我的问题"⚠️ 能力描述不清晰，关键差异化主张缺失
客户服务自助已客户自助完成文档、API、固件、报修等需求❌ 文档杂乱，无权限体系，FAQ 近乎空白
内容运营阵地新闻、案例、FAQ 持续积累，辅助 SEO 和销售❌ 内容硬编码，运营无法独立维护

1.2 现有官网问题诊断

以五大职责为框架，逐一诊断现有官网的具体问题：

❌ 品牌背书——定位主张无法兑现

官网声称"跨品牌"，但内容无法证明：

  • 无兼容品牌列表页，客户无从知晓实际接入了哪些品牌
  • 合作伙伴 Logo 墙被关闭，首页品牌展示区域代码已注释，客户看不到
  • 解决方案页文案以 Neuron 网关为主角，给人"你们家自己的设备"的感知，跨品牌能力感知不到

官网声称"多场景覆盖"，但场景不完整：

  • VPP 虚拟电厂 — 产品中心有，解决方案导航里找不到
  • PQMS 电能质量 — 产品中心有，解决方案场景页没有
  • 微电网 — 定位语里有，页面里无任何内容

结果： 最核心的竞争力，客户感知不到；相关场景客户看不到价值主张，无法转化。

❌ 客户分流——无分流设计，一套内容给所有人

  • 首页无任何针对不同客户类型的路径引导，所有人进入后看同一套内容
  • 集成商、研发型企业、VPP 运营商诉求完全不同，但官网没有分流入口
  • 没有独立的开发者中心，研发型客户无处找 API 接入信息

❌ 技术能力证明——关键差异化主张完全缺失

官网从未向客户说清楚一个关键事实：网关是可选的。

  • 云端直连同样可以接入多品牌设备，但官网只呈现了"买网关"这一条路
  • 纯云端客户认为自己不在目标客户范围内，直接流失
  • 有"数据不上云"需求的工业客户找不到本地化方案描述，流失
  • 产品与解决方案边界模糊：产品页在说解决方案的事，解决方案页在卖产品，客户不清楚该看哪个

❌ 客户服务自助——帮助中心形式大于内容

  • FAQ 页面已建立，但 Footer 入口链接被注释，用户根本找不到
  • FAQ 仅有 3 条占位示例，对实际用户毫无帮助
  • 文档分类粗糙，技术文档与软件下载混在一起，找文件效率低
  • 无任何权限体系，客户专属内容与公开内容无法区分
  • 联系方式为占位内容（400-888-8888），无法实际使用

❌ 内容运营阵地——内容硬编码，运营无法独立维护

  • 所有页面文案、FAQ、新闻动态全部硬编码在代码文件中，每次修改需开发介入、走 CI/CD 流程
  • 新闻动态只有 1 条（广交会，2025-05-05），内容严重落后
  • 合作品牌信息更新困难，干脆关闭展示模块
  • SEO 元数据硬编码，运营无法独立配置；sitemap 为静态文件，新增页面后无法自动更新

1.3 重构的本质锚点

现有定位语没有错，问题是没有用正确的内容结构去支撑它。重构不是推翻定位，而是把三个主张真正兑现在每一个页面上，同时补全官网缺失的商业骨架：

问题重构方向
跨品牌无法证明建立兼容生态页；恢复 Logo 墙；全站文案突出"品牌无关"
软硬一体没说清明确"网关可选"：云端直连 or 网关局域网，两种路径并列呈现
场景不完整补全 VPP / 电能质量场景页；产品说"能力"，解决方案说"场景+问题+方案"
无客户分流首页三路分流 CTA；新增开发者中心；联系表单按意向类型分发
自助能力缺失重建帮助中心；建立权限体系；个人中心
内容无法运营全站内容 CMS 化；SEO 可配置；sitemap 自动更新

二、官网定位升级

2.1 定位演进：继承 + 补全

旧定位并没有错，它准确描述了旭衡的技术能力与产品覆盖范围。问题是它是一个以产品为视角的自我描述，缺少两个维度：一是"谁是目标客户"，二是"通过什么商业路径触达他们"。

旧定位（产品视角，仍然成立）：
  跨品牌能源管理解决方案的优质选择
  EMS + 边缘计算网关 + 云平台
  覆盖微电网、户用、充电站、工商储

↓ 在此基础上补全

新定位（增加客户视角与商业路径）：
  在原有跨品牌、多场景能力基础上，
  面向四类目标客户，提供清晰的商业路径与分流入口：
  集成商拿整套方案 / 研发企业调 API / 硬件集成商做本地接入 / 有定制需求的客户做 VPP/BMS 定制

旧定位是官网的内容基础，新增的是商业骨架。两者共同构成重构后的完整定位。

2.2 完整定位表述

跨品牌能源管理解决方案的优质选择。
提供从整套 EMS 系统、云端 API 接入到定制化 VPP 平台的多路径服务，云端直连与网关接入灵活选择，覆盖户用、工商储、充电站、虚拟电厂全场景。

2.3 四类目标客户 × 核心诉求 × 官网路径

旭衡面向四类目标客户，每类客户的诉求和进入路径不同：

① 集成商 / 项目方
核心诉求：整套解决方案、Demo 账号、系统报价
进入路径：首页 → 解决方案  |  转化动作：申请 Demo / 联系商务
② 研发型能源企业
核心诉求：API 接口文档、协议适配、测试环境
进入路径：首页 → 开发者中心  |  转化动作：申请 API Key
③ 硬件集成商 / 工业项目方
核心诉求：网关产品采购、本地 API 对接、上位机开发
进入路径：首页 → 产品中心 → 网关产品页  |  转化动作：联系商务采购
④ VPP / BMS 定制需求方
核心诉求：定制化 VPP 平台、BMS 字典接入、白标系统
进入路径：首页 → 联系我们（定制服务直接沟通）  |  转化动作：联系我们

说明： VPP/BMS 定制需求方的路径不经过 VPP 解决方案场景页。VPP 场景页面向的是"采购标准 VPP 调度平台服务"的客户（VPP 运营商、能源聚合商）；而 VPP/BMS 定制是需要深度定制开发的服务，属于单独的商业线，由商务直接跟进，不适合通过标准场景页转化。

2.4 四大商业线 × 官网模块映射

商业线核心受众官网承载模块
成套系统解决方案集成商解决方案（HEMS / ESS / EVCMS / VPP 场景页）
云端 API 开放平台研发型企业开发者中心（新建）
纯硬件 + 本地工具工业局域网项目、硬件集成商产品中心 → 网关产品页 → 文档下载
VPP / BMS 定制开发设备厂商、代理商、有定制需求的企业联系我们（意向分类：VPP/BMS 定制化开发）

说明： VPP/BMS 定制开发不在解决方案场景页承载，因为定制服务需要一对一沟通需求，标准场景页无法有效转化。官网在「联系我们」表单中提供「VPP / BMS 定制化开发」意向选项，由产品团队直接跟进。

2.5 关键差异化主张（全站需贯穿）

接入方式由你决定，没有绑定关系。

两种接入模式，适合不同客户诉求：

接入方式适合谁核心价值
☁️ 云端直连（无需网关，API 对接）研发型企业、VPP 运营商、能源聚合商无硬件成本，快速接入，适合纯软件平台和远程调度场景
🔗 网关局域网（Neuron 网关部署现场）硬件集成商、工业项目方、对数据安全敏感的客户数据不上云，本地实时控制，适合工业局域网和隐私敏感场景

集成商 / 项目方两种模式均可按项目需求选择，也可混合使用。
这一主张在现有官网完全缺失，是重构最重要的内容补充。全站所有页面（首页、解决方案、产品页）在介绍接入能力时，都应以此为基础展开，而非默认"客户需要买网关"。

三、信息架构 IA

3.1 产品 vs 解决方案：分层逻辑

在展开导航结构之前，先明确两个频道的内容职责，这是本次重构中最重要的结构性决策。

产品中心 — 回答"旭衡卖什么"

以旭衡为主语，描述产品本身的能力、规格与功能。
目标读者：已有明确采购意向、正在做技术评估的人（采购经理、研发工程师）。
核心内容：功能列表、产品规格、截图、接口文档、下载手册。

解决方案 — 回答"客户的问题是什么、我们怎么解决"

以客户业务场景为主语，描述客户面临的处境与痛点，再给出产品组合方案。
目标读者：仍在探索阶段、需要判断"你们能解决我的问题吗"的业务决策者。
核心内容：场景描述、痛点提炼、方案架构图、产品组合（以方案为主角，产品作为手段出现）、成功案例。

两个频道的核心差异：

维度产品中心解决方案
主语这个产品这类客户 / 这个场景
核心问题它能做什么？规格是什么？我的问题能被解决吗？怎么解决？
产品出现方式产品本身是主角产品是解决手段，按需组合
CTA下载手册、查看参数、申请 Demo联系我们、获取方案报价

同一能力在两个频道的表达方式示例：

产品中心（EVCMS 云平台产品页）：
  "支持静态负载均衡与动态负载均衡两种调度模式，
   可同时管理多品牌充电桩，兼容 OCPP、私有协议等主流协议。"

解决方案（充电站场景页）：
  "停车场主线容量有限，装了 20 个充电桩却经常跳闸？
   Neuron III 网关实时检测总负载，EVCMS 智能分配各桩功率，
   不扩容、不跳闸，20 个桩全部可用。"

4.2 解决方案页（5个场景）

解决方案板块以客户业务场景为核心，每个场景页面完整覆盖：场景背景与痛点 → 方案架构与接入方式 → 核心能力 → 应用案例。产品作为解决手段出现，而非主角。

重构后官网共规划 5 个场景页：

场景一：户用家庭储能

场景背景：
家庭用户已安装光伏 + 储能系统，但各设备品牌不同、协议各异，缺乏统一的监控与调度入口；峰谷电价政策复杂，手动管理无法最大化节电收益。

核心痛点：

  • 品牌各用各的 App，信息分散，无法全局掌控
  • 充电桩、储能、光伏协同调度依赖人工，效率低
  • 无法根据实时电价自动切换用电策略

方案描述：
通过云平台或 Neuron 网关统一接入家庭全部能源设备，实现自发自用、经济模式、应急模式一键切换，结合 App 提供实时能流可视化与电费报告。

接入方式： 云端直连（品牌云 API 直接对接）/ Neuron 网关局域网（品牌不支持云 API 时）

案例方向： 中国台湾别墅小区；澳洲家庭储能用户（结合当地电价结构）

场景二：工商业储能

场景背景：
工厂、商业楼宇、园区面临峰谷电价差大、用电成本高的压力，储能系统作为核心降本手段，但分布式多站点管理复杂、跨品牌设备接入难。

核心痛点：

  • 分布式站点各自孤立，无统一监控中心
  • 不同品牌 PCS、BMS 协议不统一，集成成本极高
  • 峰谷套利策略依赖人工设置，无法动态响应电价波动

方案描述：
通过 Neuron 网关或云端 API 接入多品牌储能设备，ESS 云平台实现跨站点统一监控、峰谷自动套利、告警分级管理、能效报告。

接入方式： Neuron 网关（现场设备协议转换）/ 云端直连（品牌云 API 对接）

案例方向： 深圳龙华分布式储能项目（5 天完成云平台搭建，10 天完成跨品牌设备接入）

场景三：充电站负载管理

场景背景：
停车场、办公楼、企业园区需要部署多台充电桩，但主线电容有限，多桩同时快充极易触发主断路器跳闸；充电桩品牌混杂，协议不一，统一管理困难。

核心痛点：

  • 受限于主线容量，充电桩无法满功率使用
  • 多品牌充电桩各自独立，缺乏整体调度
  • 无法按优先级智能分配充电功率（如 VIP 优先）

方案描述：
Neuron III 网关实时检测主线总负载，配合 EVCMS 云平台动态分配各桩可用功率，实现在不扩容主线的前提下多桩协同充电，支持平均模式、排队模式、VIP 模式。

接入方式： Neuron III 网关（PLC/RS485 本地控制，实时毫秒级响应）

案例方向： 深圳罗湖办公楼停车场；马来西亚充电站项目

场景四：VPP 虚拟电厂调度

场景背景（结合澳洲市场）：
澳大利亚等地屋顶光伏与家庭储能渗透率全球领先，但分散的能源资产无法单独参与电网调度。VPP 通过聚合大量分布式电池，将其统一协调为一座"虚拟电厂"，在电网用电高峰时集中放电，在低谷时集中储能，VPP 运营商可从电力现货市场、需求响应机制（FCAS）中获取收益，并将收益部分返还给资产所有者。
澳洲主流 VPP 模式（Origin、AGL、Tesla Energy 等）的核心技术逻辑是：通过接入各品牌家庭储能设备的云端 API，实时获取电池 SOC（荷电状态）、可用功率，并下发充放电指令，整个调度链路完全在云端完成，无需现场硬件介入。

旭衡的 VPP 定位：
提供面向 VPP 运营商的底层调度平台。核心能力是**通过接入第三方品牌储能设备的云端 API，统一实现对多品牌真实电池的充放电调度**，而非模拟仿真。旭衡不做电力交易，聚焦设备侧的协议接入与调度执行层。

核心痛点：

  • 不同品牌储能系统 API 协议各异（Modbus、私有 REST、MQTT），接入成本极高
  • 自研调度平台技术门槛高、周期长，难以快速进入市场
  • 需要支持实时响应（秒级调度指令下发），可靠性要求高

方案描述：
旭衡 VPP 平台已打通多个主流品牌储能设备的云端 API，VPP 运营商只需对接旭衡统一调度接口，即可向目标设备发出充放电指令、查询实时 SOC、订阅告警事件。平台支持批量设备同时调度，可对接上游电网信号或自定义调度策略。

技术亮点：

  • 多品牌云 API 适配层：已接入 X 个品牌储能设备
  • 统一调度指令集：标准化充电 / 放电 / 待机 / 紧急停止指令
  • 实时状态订阅：SOC、功率、告警状态毫秒级推送
  • 调度记录与审计：完整的指令下发与执行记录

接入方式： 云端直连（主要模式，通过品牌 API 调度实际电池）

适用客户方向： VPP 运营商、能源聚合商、参与需求响应项目的能源服务公司

场景五：电能质量远程监控

场景背景：
工业园区、数据中心、医院、商业综合体对电能质量要求严苛，谐波污染、电压骤降、三相不平衡等问题直接影响设备寿命与生产安全，但传统人工巡检模式响应慢、覆盖不全。

核心痛点：

  • 谐波问题隐蔽，设备异常时才发现，损失已经发生
  • 多个监测点数据分散，无统一分析中心
  • 补偿设备（SVG、APF）参数调整依赖现场工程师，无法远程操作

方案描述：
通过 PQMS 平台实时采集各监测点的电压、电流、谐波、功率因数等指标，地图化展示站点健康状态，异常自动预警并推送通知，支持远程调整补偿参数，并提供历史分析报告帮助客户定位电能质量劣化根因。同时支持远程 OTA 升级现场设备固件，无需现场运维。

接入方式： Neuron 网关（本地采集，支持 Modbus / RS485 仪表接入）/ 支持品牌云 API 直接对接

案例方向： 工业园区谐波治理；数据中心电能质量管控

4.3 兼容生态页

这是展示技术壁垒、建立客户信任的核心差异化页面。

设计背景说明（为什么要做能力标签）：
在能源 IoT 领域，「接入」不等于「全功能控制」。很多品牌的云端 API 只开放遥测（Telemetry）权限，即设备状态数据的只读上报，不开放充放电控制（调度指令下发）权限。
集成商如果只看"是否已接入"，签了合同才发现某品牌只能看数据、无法控制充放电，会产生严重客诉和返工成本。
因此，兼容生态页必须在品牌展示中明确区分：
- 遥测（只读）：可实时读取设备 SOC、功率、温度等状态数据
- 充放电控制（调度）：可下发充电 / 放电 / 待机等控制指令
- 历史数据导出：可查询并导出历史运行数据
这一信息在客户选型阶段就明确传达，能大幅减少预期差异和售后纠纷。

版本兼容性矩阵（新增）：

为什么需要这个？
IoT 硬件的固件版本、云平台版本、配置工具版本之间存在依赖关系。客户如果只看到一个固件下载列表，不知道该版本是否兼容自己当前的云平台，极易刷错固件导致设备无法启动（俗称"变砖"），引发大量售后投诉。版本兼容性矩阵是 IoT 行业帮助中心的标准配置，在同类产品中普遍存在。

IoT 设备固件与软件之间存在版本依赖关系，客户下载固件前必须能快速确认版本兼容性，避免刷错固件导致设备异常。

5.1 各模块通用规范

  • 多语言支持：所有面向用户的内容字段，均需支持多语言维护。核心维护语言为简体中文（zh-CN）和英文（en-US），繁体中文（zh-TW）由 zh-CN 自动转换，德/法/葡/西/俄五种语言以 en-US 为源、机器翻译辅助、人工校对后发布

  • 支持版本号管理，新版本上传后旧版本仍可保留归档
  • 记录每个文件的下载次数，了解用户关注点
  • 文件上传后自动生成下载链接，无需手动配置
  • 固件文件支持填写版本依赖关系（依赖的云平台最低版本、配置工具最低版本），用于帮助中心版本兼容性矩阵展示及下载时的联动提示

  • 保存后官网自动更新，无需重新部署

5.6 客户线索管理需求

集中管理官网所有表单提交，包括业务咨询、在线报修、API Key 申请、Demo 申请。核心需求：

  • 统一收件箱展示所有来源的线索，记录提交时间、来源页面、意向分类
  • 每条线索支持状态流转：待处理 → 跟进中 → 已关闭
  • 新线索提交后自动推送通知至企业微信对应业务群（按意向类型分发）
  • 支持按时间、状态、意向类型筛选
  • 预留标准化 CRM Webhook/API 出口：表单提交后，除企微通知外，可将线索数据推送至外部 CRM 系统，推送字段包含来源页面、意向分类、提交时间，实现营销转化闭环

什么是 CRM？
CRM（客户关系管理系统）是用来统一管理销售线索、记录客户沟通历史、追踪成单进度的工具。企微群消息通知只是"提醒"，不是"沉淀"——线索多了之后，没有 CRM 就无法知道有多少线索在谈、多少丢单、哪个渠道转化率最高。常见工具：国内有纷享销客、飞书 CRM；海外有 HubSpot、Salesforce。
官网侧只需预留标准 Webhook 出口，CRM 工具由业务团队按实际需求自行选型接入，不在本次重构范围内。

5.10 多语言字典管理需求

管理官网 UI 界面的公共翻译词条（按钮、标签、提示文字、导航文字等固定文案），让运营人员在后台统一维护，不写死在代码里。

为什么需要字典管理？
官网 §7.3 要求支持 8 种语言，除了页面内容需要多语言维护，UI 界面上的公共固定文案（如"查看更多"、"联系我们"、"暂无数据"等）也需要各语言版本。如果这些词条写死在代码里，每次调整措辞或新增语言都要走代码发布流程。字典管理让运营人员直接在后台修改、新增语言，无需开发介入。

核心能力：后台添加语言 → 前台自动支持
在「语言管理」中新增一种语言（如 de-DE），系统自动：① 在前台语言切换器中出现该语言选项（发布后可见）；② 在 CMS 所有内容编辑界面增加该语言的编辑 Tab；③ 导出待翻译词条清单。整个过程不需要修改任何代码。

核心需求：

  • 支持管理官网所有语言的 UI 公共词条（按 key-value 对维护）
  • 词条按语言分组维护，核心语言（zh-CN / en-US）人工维护；zh-TW 由 zh-CN 自动转换；de-DE / fr-FR / pt-BR / es-ES / ru-RU 由 en-US 机器翻译初稿，支持人工校对覆盖
  • 支持新增语言：当官网需要扩展新语言时（如 de-DE），在后台添加语言后即可开始维护词条，无需修改代码结构
  • 支持批量导入/导出（Excel/CSV），方便交给翻译人员处理后批量回填
  • 词条修改后官网实时生效，无需重新部署
  • 支持搜索词条 key 或内容，快速定位需要修改的词条

5.12 媒体库（图片 / 视频统一管理）

  • 统一上传入口：运维人员在媒体库上传图片/视频，系统自动上传至 OSS 并生成 CDN 加速链接，无需手动配置
  • 图片分类与标签：支持按用途分类（产品图 / Banner / 案例封面 / Logo / 其他），支持搜索文件名
  • 图片替换：替换某张图片时，上传新图后可选择「覆盖原链接」（所有引用该图片的页面自动更新）或「新建链接」（保留旧图，手动更新引用）
  • 图片裁剪与压缩：上传时自动压缩（WebP 格式优先），支持在线裁剪以适配不同展示尺寸（如 Banner 16:9、产品图 1:1、缩略图 4:3）
  • 引用追踪：每张图片显示被哪些页面/内容引用，避免误删正在使用的图片
  • 软删除：删除图片前检查引用，有引用时禁止删除并提示；无引用时允许软删除，保留 30 天后物理删除
  • 容量统计：显示已用存储空间，便于运维监控 OSS 成本

与其他模块的关系：富文本编辑器（新闻、FAQ、解决方案内容）插入图片时，均调用媒体库选择器，而非直接上传。保证所有图片资产统一管理，不散落在各编辑器内。

旭衡业务覆盖多个海外市场（包括但不限于澳洲、东南亚、拉美等），官网需要区分两个层次：

  • 层次一：多语言（i18n） — 文字翻译
    核心维护语言：zh-CN / en-US（人工全量维护）；zh-TW 自动转换；de/fr/pt/es/ru 机器翻译辅助。语言切换对所有用户可见，通过 URL 或语言选择器控制（详见 §7.3）。
  • 层次二：多区域（Localization） — 内容差异化
    同一语言下，不同市场的用户看到针对性的场景描述、案例和法规话术。CMS 需支持按「语言 + 区域」的组合管理内容，而不仅是按语言。具体覆盖哪些区域、何时上线，由运营按市场进度按需配置，PRD 不约束。
为什么翻译不够？

旭衡业务已覆盖澳洲、巴西、俄罗斯等多个市场。同样是英文版官网，澳洲客户关心当地电力市场的调度机制，东南亚客户关心充电桩和工商储能，用同一套英文内容打全球是不够的。Localization（本地化）的核心是：相同语言下，不同市场看到的场景描述、案例和合规话术可以不同。

7.10 隐私合规（GDPR / LGPD）

为什么要做这个？

GDPR 是欧盟的数据隐私法规，LGPD 是巴西的对应法规，两者均要求网站在收集用户数据前必须明确获得授权，违规最高罚款可达年营收的 4%。旭衡官网收集用户邮箱、公司信息、设备坐标，并向海外用户提供 API 服务，属于合规要求的覆盖范围。大型能源企业在采购前会做供应商合规审查，官网不合规会直接影响商务谈判。

隐私合规为 P0 必须完成项，上线前需由法务确认合规文本，不得使用模板占位。

旭衡业务覆盖欧盟及巴西等有严格数据隐私法规的市场，官网必须满足以下合规要求：

  • Cookie 授权管理：首次访问时展示 Cookie 授权弹窗，支持分类授权（必要 / 分析 / 营销）。用户可随时撤回授权，设置入口在页脚常驻。按访客地区自动适配合规标准（欧盟用户遵循 GDPR，巴西用户遵循 LGPD）。
  • 表单合规：「申请接入」「联系我们」「在线报修」等所有收集用户信息的表单，底部必须包含：《隐私政策》链接（CMS 可配置，支持多语言版本）、《数据处理同意声明》勾选框（必填，不勾选无法提交）、若涉及数据跨境传输需补充《数据出境合规声明》。
  • 隐私政策页：独立页面，通过 CMS 管理内容，支持多语言版本独立维护。涵盖：数据收集范围、使用目的、存储期限、用户权利（查阅/删除/更正）、数据出境说明。

7.11 访问加速与国内备案

为什么要双节点？

旭衡官网需要同时服务国内客户（访问速度快、合规要求备案）和海外客户（不能因国内网络波动影响固件下载和 API 调用）。单节点无法同时满足两端需求，因此需要国内外分别部署加速节点。ICP 备案是国内网站上线的法律要求，包含中文内容的站点必须完成后才能对国内用户提供服务。

旭衡官网同时服务国内和海外用户，且包含大量文件下载（文档/固件/软件），需满足以下访问要求：

  • 国内访问：域名（alwayscontrol.com.cn）需完成 ICP 备案。国内用户访问须经国内加速节点，保证访问速度；文件下载走国内对象存储 + CDN，避免境外带宽影响体验。
  • 海外访问：海外用户访问须走境外加速节点，确保不受国内网络波动影响。API 文档、固件下载等资源需在海外节点缓存，开发者调用 api.alwayscontrol.com.cn 的延迟应在可接受范围内。
  • 智能路由：须支持按用户地理位置自动路由至就近加速节点，海外用户不走国内节点。
  • 备案要求：zh-CN 版本内容必须在 ICP 备案完成后才能上线，此项为上线硬性前提。

具体 CDN 服务商选型、部署架构详见技术选型文档。

8.1 设计原则

官网采用 RBAC + 资源标签（Role-Based Access Control + Resource Tag） 混合模式：

  • 角色决定基础访问层级（是否登录、是否为客户）
  • 资源标签决定具体内容的可见性（购买了哪条产品线）
  • 两者取交集，精准控制每个客户能看到什么

这套模型参考了 Twilio、Salesforce 等主流 B2B 平台的权限设计，适合产品线多、客户群体差异大的场景。
