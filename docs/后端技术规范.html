层级要求备注

运行时Node.js ≥ 20 LTS
数据库PostgreSQL ≥ 15不使用 SQLite（仅限开发环境）

二、API 设计规范

基础规则

  • RESTful 风格，JSON 格式
  • 所有接口添加 /api 前缀
  • 版本化：/api/v1/...
  • 使用 HTTPS，HTTP 强制跳转

四、文件上传规范

上传流程


前端 → POST /api/v1/media/presign → 获取 OSS 预签名上传 URL
前端 → PUT {presign_url} → 直传 OSS（不经过 CMS 服务器）
前端 → POST /api/v1/media/confirm → 通知 CMS 记录文件元数据
CMS  → 自动生成 CDN 访问链接存入数据库

限制

文件类型最大大小允许格式
图片10MBjpg / png / webp / svg
文档100MBpdf / docx / xlsx / zip
固件500MBbin / hex / zip
软件安装包500MBexe / dmg / apk / zip

图片处理

  • 上传后自动生成 WebP 版本
  • 生成三种尺寸：original / medium(800px) / thumbnail(200px)
  • 记录所有尺寸的 CDN URL

五、邮件发送规范


// 表单提交后触发（异步队列，不阻塞响应）
await emailQueue.add('send-notification', {
  to: config.internal_emails,
  template: 'lead-notification',
  data: { lead, sourceUrl }
})

await emailQueue.add('send-user-receipt', {
  to: lead.email,
  template: `user-receipt-${lead.type}`,
  locale: lead.locale,
  data: { lead }
})

规则：

  • 邮件发送必须异步处理（队列），不在请求响应链中同步发送
  • 发送失败重试 3 次，间隔指数退避
  • 记录发送状态到 leads.user_email_sent_at
  • 敏感信息（SMTP 密码）只通过环境变量注入

六、安全规范

类型要求
CORS只允许官网域名（alwayscontrol.com.cn / staging 域名）
速率限制表单提交接口：同 IP 每分钟最多 5 次
输入校验所有输入字段服务端校验，不依赖前端校验
XSS富文本内容存储前经 DOMPurify 过滤
SQL 注入使用 ORM（TypeORM / Knex），不拼接 SQL 字符串
文件类型上传前检查 Magic Bytes，不只信任扩展名
敏感日志日志中不输出邮件地址、手机号等 PII
HTTPS所有接口强制 HTTPS，HSTS 头部

八、部署要求

  • Node.js 进程由 PM2 守护，崩溃自动重启
  • 数据库连接池大小：生产 ≥ 10，Staging ≥ 3
  • 日志输出到文件，保留 30 天，ERROR 级别触发告警
  • 生产数据库每日自动备份，保留 7 天
  • Staging 数据库可一键从生产快照恢复（用于测试数据同步）

九、数据流通设计

表单提交流（联系我们 / 在线报修）


① 用户提交表单
    ↓
② POST /api/v1/leads（或 /api/v1/repair-requests）
    ↓  服务端校验（必填字段 + 格式 + rate limit）
    ↓  写入数据库（leads 表）
    ↓  立即返回 201 成功响应（不阻塞后续通知）
    ↓
③ 异步队列并行触发（最长5分钟内完成）：
    ├─ A. 企微 Webhook
    │     根据 lead.type 从 notification_configs 查对应 webhook_url
    │     POST Webhook → 企微「对应群」
    │
    ├─ B. 内部通知邮件
    │     根据 lead.type 查对应内部邮箱列表
    │     发送邮件（含完整表单数据）
    │
    └─ C. 用户回执邮件
          根据 lead.locale 选择邮件模板
          发送至 lead.email

④ 任意通知失败：重试3次（指数退避），失败写 error log
   不影响已返回的 201 响应（用户侧无感知）

CMS 内容发布流（内容更新 → 前端生效）


① 运营在 CMS 后台修改内容并点击「发布」
    ↓
② CMS 触发 Webhook（配置在 CMS 后台）
    POST /api/v1/revalidate
    Header: Authorization: Bearer {REVALIDATE_SECRET}
    Body: { "type": "product", "slug": "neuron-ii", "locale": "zh-CN" }
    ↓
③ Next.js ISR 缓存刷新
    revalidateTag(type)  或  revalidatePath(path)
    ↓
④ CDN 边缘节点缓存失效
    ↓
⑤ 下次用户请求时，服务端重新从 CMS 拉取最新数据渲染页面

规则：
- 全量刷新（type=all）仅在 CMS 后台「缓存管理」手动触发，不自动全量
- 每次发布精确传入 type + slug，避免不必要的全量缓存失效
- REVALIDATE_SECRET 通过环境变量注入，不写入代码库
