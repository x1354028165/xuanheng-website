# Coding Agent 指令模板

> 每次调度 Claude Code 执行编程任务时，以此为基础填写。
> 框架固定，内容按任务替换。

---

## 模板结构

```
[任务标题]：[一句话描述做什么]

工作目录：[绝对路径]

【黄金铁律】
- [不可违背的约束，例如：不修改 M1~M7 任何代码]
- [不可违背的约束，例如：所有现有 XX 个测试必须维持全绿]
- [其他约束...]

【手术前必读】
先完整阅读以下文件，理解现有架构风格后再动手：
1. [文件路径] — [说明这个文件的作用]
2. [文件路径] — [说明这个文件的作用]
3. [文件路径，尤其是现有测试文件] — [了解测试风格和 mock 模式]

==============================
步骤 1：[步骤名称]
==============================

[具体到：文件路径、函数签名、关键逻辑、边界条件]

⚠️ 若涉及「硬编码 → 动态配置」类型重构：
- Fallback 值必须与原硬编码数值完全一致
  示例：原代码 MIN_SOC = 20，则 DEFAULT_CONFIG = { minSoc: 20 }
  目的：保护现有测试，避免数值变动导致测试失败

==============================
步骤 2：[步骤名称]
==============================

[具体内容...]

==============================
步骤 3：撰写/更新测试
==============================

⚠️ 测试 Mock 注意事项（Jest）：
- global.fetch mock 或其他模块级 mock 必须在所有 import 之前声明
  正确写法：
    // 必须在 import 之前
    const mockFetch = jest.fn();
    global.fetch = mockFetch;
    
    import { handler } from '../src/xxx';  // import 在后面
    
- beforeEach 中 mock 的返回值应与原硬编码一致，保护现有测试不因数值变动而失败

[具体测试覆盖要求：
  - 正常路径
  - 边界条件（如无效参数、越界值）
  - 错误处理（如外部服务不可用时的 fallback）
  - 必须验证的关键行为...]

==============================
步骤 4：验收
==============================

cd [工作目录]
[测试命令，如 npm run test]

预期：[原有 XX 个测试 + 新增 YY 个测试] 全部通过
如有失败：阅读错误信息，修复后重新运行，不要跳过

==============================
步骤 5：提交
==============================

git add -A
git commit -m "[feat/fix/refactor](模块): 一句话描述变更内容"
git push origin [分支名]

回报：
- 新增/修改了哪些文件
- 测试总数（原有 + 新增）
- commit hash

==============================
完成通知（必须执行）
==============================

完成后执行：
openclaw system event --text "Done: [任务标题] — [简要结果]" --mode now
```

## 使用说明

**填写顺序：**
1. 先写黄金铁律（约束边界）
2. 列出必读文件（让 agent 先理解现有风格）
3. 按步骤写具体要求（越具体越好：函数签名、字段名、SQL 结构）
4. 写测试覆盖要求（特别注明关键防弹测试）
5. 结尾永远是：npm test 验收 → git push → openclaw 通知

**什么时候需要补充规格代码（如 SQL、TypeScript interface）：**
- 当 agent 需要创建新的数据结构时：直接给出字段定义
- 当有跨文件命名约定时：给出示例让 agent 对齐风格
- 当有版本兼容性要求时：明确给出约束

**什么时候不需要补充代码：**
- 修改现有逻辑（让 agent 先读现有代码，风格自然对齐）
- 简单增删字段（描述清楚即可）
